Full transparency into what our AI can do, what it can't, and how every decision is made. No black boxes.

Core Components

System Architecture

📋

12-Rule Fraud Engine

Deterministic rule-based checks that evaluate known fraud patterns.

• Amount vs Policy Limit check
• Claim frequency analysis
• Round number detection
• Late reporting flags
• Prior fraud history
• OCR mismatch detection
✅ Fully transparent & explainable
🤖

ML Fraud Model

Scikit-learn classifier trained on claim data with 11 numeric features.

• Model: Random Forest Classifier
• Features: 11 numeric inputs
• Output: Fraud probability (0-100)
• Blended 40/60 with rule score
• Retrained on CSV dataset
⚠️ Simple classifier, not deep learning
📄

OCR Document Processing

Text extraction using pdfplumber (PDFs) and Pillow/pytesseract (images).

• PDF text extraction via pdfplumber
• Image OCR via pytesseract (if available)
• Regex-based field extraction
• Amounts, dates, names, references
• Basic tampering heuristics
⚠️ Pattern matching, not document AI
Honest Disclosure

Capabilities & Limitations

✓Checks 12 deterministic fraud rules with full transparency

✗No actuarial modeling for settlement calculations
Processing Flow

6-Stage Processing Pipeline

📋

Stage 1: Claim Submitted

Claim is received, validated, and queued. Reference number generated. Basic field validation performed.

📄

Stage 2: Document Verification

OCR extracts text from uploaded documents. Regex patterns parse amounts, dates, references. Checks required document completeness per claim type. Limitation: Cannot visually analyze images or detect sophisticated forgeries.

🔍

Stage 3: Fraud Analysis

12 deterministic rules check for known fraud patterns (amount thresholds, timing, frequency). ML model predicts fraud probability using 11 features. Final score = 60% rules + 40% ML. Limitation: Rules are static, ML model is a simple classifier.

📊

Stage 4: Risk Scoring

Eligibility check (amount limit, policy status, claim history). Settlement calculated with deductibles and risk adjustments. Limitation: Uses fixed deductible rates, not actuarial models.

⚖️

Stage 5: Decision Engine

STP gate logic: if eligible + docs OK + fraud < 30 → auto-approve. Fraud ≥ 70 → manual review. Not eligible → reject. Simple decision tree, not learned.

💸

Stage 6: Settlement

Final status set (approved/rejected/manual_review). Explainability report generated with template-based text. Payout status updated. Audit trail finalized.

ML Model Details

11-Feature Input Vector

claim_amountfloat
claim_to_limit_ratiofloat
prior_claimsint
prior_fraud_flagsint
days_since_incidentint
policy_age_daysint
is_round_number0/1
is_weekend0/1
document_countint
ocr_mismatch0/1
late_reporting0/1

Score Blending Formula

final_score = (rule_score × 0.6) + (ml_score × 0.4)

Decision Thresholds

Score ≤ 30 → Auto-Approve
Score 31-69 → May Approve / Review
Score ≥ 70 → Manual Review Required

Explainability Approach

Explanations are generated using template-based text that maps rule results and scores to human-readable factors. This is NOT model introspection (SHAP/LIME) — it's structured reporting of the decision inputs.
